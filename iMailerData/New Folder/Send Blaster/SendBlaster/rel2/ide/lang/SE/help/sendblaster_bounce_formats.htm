Anpassade format för Bounce-backs

Format för Bounce-backs var först designade för att kunna läsas av människan, inte av maskiner, men ändå finns det inga standarder eller regler. SendBlaster kan hantera de flesta formaten för bounce-backs med hjälp av sin interna databas, men du kanske stöter på något format som inte känns igen av mjukvaran. Om det händer, så kan du manuellt lägga till ett nytt anpassat format till SendBlaster's databas för bounce-backs.

Ett anpassat format består av ett antal regler; alla måste följas för att ett meddelande ska kunna ses som en bounce-back (med andra ord, den logiska operanden för reglerna är AND (och)).

När du vill lägga ttill ett  Nytt format måste du först av allt identifiera mönster i ämnesraden eller i headern, vilket unikt identifierar meddelanden som ett studsat meddelande (en Hård studs , om du kryssar i rutan, eller en mjuk om den är inte är ikryssad). Dropplistan över headers är bara ett exempel, du kan manuellt lägga till specifika headers genom att skriva in headernamnet i Header -fältet och det önskade mönstret i Värde-fältet.

Ett  mönster inkluderar wildcards (tecknet "*" vilket betyder "vilket tecken som helst"): om du t.ex. skriver "*delivery*notification*" som ämnesmönster så kommer alla meddelanden som innehåller "delivery" och "notification" i ämnet - t.ex. "delivery to xxx@yyy.com notification failure" - att ses som ett studsat meddelande, förrutsatt att det matchar de andra reglerna som du kan ha definierat. Om du använder "delivery notification", utan wildcards, kommer bara meddelanden som har den exakta frasen "delivery notification" att hanteras som studsade meddelanden - "delivery to xxx@yyy.com notification failure" skulle inte ses som ett giltligt ämne.

Du kan också skriva in mer specifika ord (separerade med komma) som måste finnas i kroppen av meddelandet: om inget av dessa ord hittas hanteras inte meddelandet som ett studsat. Som ett exempel, de flesta bounce-backs innehåller ordet "orsak" ("reason", följt med text som förklarar varför meddelandet inte levererades). Vänligen notera att listan över ord inte är ett mönster (du kan inte använda wildcards) och den logiska operanden är OR (eller).

Om ett meddelande möter alla regler som du har definierat, kommer det att hanteras som ett studsat meddelande och alla epostadresser som hittas i meddelandet sägs upp, om de också finns i listan som du laddar ner studsar för. Du kan också välja att exkludera typiska postmaster-adresser med mönster.

     

Ett exempel

Låt oss säga att du vill att meddelanden ska hanteras som bounce-backs när de:

a) har orden "undeliverable" (ungefär "olevererbart") och "message" ("meddelande") i ämnet;
b) kommer från en "postmaster@..."-adress;
c) Content-Type-headern är "delivery notification" eller "delivery failure notification" ("leveransnotifikation" eller "leveransfelsnotifikation")
d) innehåller orden "reason" eller "error" ("orsak" eller "fel") i kroppen

och du vill givetvis inte att avsändarens ("postmaster@...") adress ska bli uppsagd, eftersom den riktiga adressen finns någon annanstans i kroppen.

Såhär skulle formatet kunna se ut:

Hård studs:   (checked)
Ämnesmönster:   *undeliverable*message*
1. Header:   From
  Värde:   *postmaster*
2. Header:   Content-Type
  Värde:   *delivery*notification*
Kroppen innehåller :   reason, error
Exkludera :   postmaster@*

Och här är ett mycket enklare format, vilket hanterar alla meddelanden som har ordet "undeliverable" i ämnesraden (och exkluderar både postmaster@... och majordomo@...):

Hård studs :   (checked)
Ämnesmönster:   *undeliverable*
1. Header:   (empty)
  Värde:   (empty)
2. Header:   (empty)
  Värde:   (empty)
Kroppen innehåller :   (empty)
Exkludera:   postmaster@*, majordomo@*
